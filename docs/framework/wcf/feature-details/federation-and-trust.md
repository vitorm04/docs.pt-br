 <span data-ttu-id="5a16f-108">Da mesma forma, ao usar chaves simétricas, as chaves são criptografadas para o serviço de destino, portanto, você deve configurar o serviço de token de segurança com as credenciais corretas para o serviço de destino; caso contrário, não será possível criptografar a chave para o serviço de destino e, novamente, o cliente não poderá se comunicar com o serviço.</span><span class="sxs-lookup"><span data-stu-id="5a16f-108">Similarly, when using symmetric keys, the keys are encrypted for the target service, so you must configure the security token service with the correct credentials for the target service; otherwise, it will be unable to encrypt the key for the target service, and again, the client will be unable to communicate with the service.</span></span>  
  
 <span data-ttu-id="5a16f-109">Os serviços WCF usam o valor da <xref:System.ServiceModel.Channels.LocalServiceSecuritySettings.MaxClockSkew%2A> Propriedade no [SecurityBindingElement](../../../../docs/framework/wcf/diagnostics/wmi/securitybindingelement.md) para permitir a distorção de relógio entre o cliente e o serviço.</span><span class="sxs-lookup"><span data-stu-id="5a16f-109">WCF services use the value of the <xref:System.ServiceModel.Channels.LocalServiceSecuritySettings.MaxClockSkew%2A> property on the [SecurityBindingElement](../../../../docs/framework/wcf/diagnostics/wmi/securitybindingelement.md) to allow for clock skew between the client and service.</span></span> <span data-ttu-id="5a16f-110">Na Federação, a `MaxClockSkew` configuração se aplica a distorções de relógio entre o cliente e o serviço de token de segurança de onde o cliente obteve o token emitido.</span><span class="sxs-lookup"><span data-stu-id="5a16f-110">In federation, the `MaxClockSkew` setting applies to clock skews between both the client and the security token service from where the client obtained the issued token.</span></span> <span data-ttu-id="5a16f-111">Portanto, os serviços de token de segurança não precisam fazer concessões de distorção de relógio ao definir os tempos de efetivação e de expiração do token emitido.</span><span class="sxs-lookup"><span data-stu-id="5a16f-111">Therefore, security token services need not make clock-skew allowances when setting the issued token's effective and expiration times.</span></span>  
  
> [!NOTE]
> <span data-ttu-id="5a16f-112">A importância da distorção do relógio aumenta conforme o tempo de vida do token emitido diminui.</span><span class="sxs-lookup"><span data-stu-id="5a16f-112">The importance of clock skew increases as the lifetime of the issued token shortens.</span></span> <span data-ttu-id="5a16f-113">Na maioria dos casos, a distorção do relógio não será um problema significativo se o tempo de vida do token for de 30 minutos ou mais.</span><span class="sxs-lookup"><span data-stu-id="5a16f-113">In most cases, clock skew is not a significant issue if the token lifetime is 30 minutes or more.</span></span> <span data-ttu-id="5a16f-114">Os cenários com vidas de vida mais curtas ou onde o tempo de validade exato do token são importantes devem ser projetados para levar em conta a inclinação do relógio.</span><span class="sxs-lookup"><span data-stu-id="5a16f-114">Scenarios with shorter lifetimes or where the exact validity time of the token is important should be designed to take clock skew into account.</span></span>
